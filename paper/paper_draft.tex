We present \textbf{ProvShield}, a provenance-typed runtime enforcement system for MCP and skill-based LLM agents. ProvShield treats the LLM as an untrusted planner and moves authorization to the runtime. Content entering the agent is assigned unforgeable sidecar provenance labels that track integrity and confidentiality. Tools are declared with effect types and sinks. Before any tool call executes, a runtime monitor checks whether the proposed action, destination, payload, and capability token satisfy a source-to-sink policy. For high-risk effects, ProvShield requires a bound user-intent bridge that is specific to the action, sink, destination, payload digest, principal, and expiration.

We formalize ProvShield with a labeled transition system and prove label unforgeability, capability-token unforgeability, no unauthorized secret exfiltration, low-integrity control prevention for high-risk effects, and bridge non-replay under the stated trusted computing base. We implement ProvShield as an MCP proxy with JSON-RPC support, skill loader with HMAC verification, context builder, policy engine, bridge manager, and audit logger. We evaluate it on 780 scenarios (530 attack, 250 benign) spanning skill injection, MCP metadata poisoning, web/email prompt injection, RAG injection, and adaptive white-box attacks against six defense configurations, including 88 workflow-embedded attack scenarios (45 strong + 22 targeted + 21 high-rate). ProvShield achieves \textbf{0.6\%} end-to-end attack success rate (95\% CI: [0.2\%, 1.7\%]) while maintaining 92.4\% benign task completion, an 88\% reduction from the no-defense baseline of 5.1\%.

\section{Introduction}
\label{sec:introduction}

LLM agents are no longer passive chat systems. They read webpages, search private documents, inspect email, call MCP tools, load skills, and perform write actions on behalf of users. This shift changes the security problem. A prompt injection does not need to compromise the model in the traditional sense; it only needs to convince the model to use the authority already granted to the agent.

The root cause is an \emph{instruction/data boundary failure}. A single model context may contain system policies, user goals, tool descriptions, retrieved documents, email text, skill instructions, and tool outputs. All are represented as natural language, but they do not have the same authority. A webpage should provide facts, not new objectives. A tool output should inform the agent, not authorize a new write. A tool description should describe the interface, not grant itself privilege. A skill file should not override system policy. Yet many agents feed these sources to the model without a runtime-enforced distinction.

A common response is to harden the system prompt: tell the model to ignore instructions in webpages and emails~\cite{openai_agent_prompt_injection_defense}. This is useful but brittle. Another response is input filtering: classify malicious text before it reaches the model. This fails when attacks are contextual, indirect, or socially engineered~\cite{greshake2023indirect}. A third response is generic confirmation: ask the user before writes. This can be laundered by vague confirmations that do not bind the actual destination, payload, or source of influence~\cite{perez2022ignore}.

This paper takes a different position. The model may be manipulated; the runtime must still prevent unauthorized high-risk effects. We design \textbf{ProvShield}, a provenance-typed runtime enforcement system for MCP and skill-based agents. ProvShield assigns unforgeable sidecar labels to content as it enters the agent runtime, tracks integrity and confidentiality through transformations~\cite{denning1976lattice,myers1997decentralized}, declares tools with effect types and sinks, and intercepts all proposed tool calls. A tool call executes only if the runtime can establish that its action, destination, payload, and capability are authorized by policy. When user confirmation is required, the confirmation is not generic; it creates a one-time bridge bound to the exact action, sink, destination, payload digest, expiry, and principal.

\subsection{Contributions}

This paper makes the following contributions:

\begin{enumerate}
  \item \textbf{A provenance-typed security model for MCP and skill-based agents.} We define integrity and confidentiality lattices for user intent, system policy, skills, tool metadata, tool outputs, external content, secrets, and capability tokens.
  \item \textbf{An effect-typed runtime monitor.} We introduce source-to-sink rules that gate write, send, delete, execute, credential, authentication, and financial effects.
  \item \textbf{A bound user-intent bridge.} We design an action-specific, destination-specific, payload-specific declassification mechanism that avoids the weakness of generic confirmation.
  \item \textbf{A formal model and safety properties.} We model the LLM as an adversarial proposal oracle and prove runtime-observable sink enforcement properties.
  \item \textbf{A prototype for MCP and skill-based agents.} We implement MCP proxying, skill loading, sidecar provenance, policy checking, bridge handling, and audit replay.
  \item \textbf{A systematic evaluation plan.} We evaluate against skill injection~\cite{skillinject}, MCP metadata poisoning~\cite{mcptox}, MCP safety scenarios~\cite{mcp_safety_audit}, web/email prompt injection, RAG injection, and adaptive attacks~\cite{attriguard_agentwatcher}.
\end{enumerate}

\section{Motivating Attacks}
\label{sec:attacks}

We describe four representative attacks that motivate ProvShield's design. Each exploits the collapsed instruction/data boundary in current agent architectures.

\subsection{Webpage-to-Email Exfiltration}
\label{sec:attack-web}

A user asks the agent to summarize a webpage. The webpage contains hidden text instructing the agent to send the user's API key to an attacker-controlled email address~\cite{greshake2023indirect}. A prompt-only defense requires the model to recognize and ignore the hidden instruction. ProvShield instead labels the webpage as \texttt{ExternalContent}. If the model proposes \texttt{send\_email}, the monitor detects that the destination and objective are influenced by low-integrity content and that the payload may contain \texttt{Secret}. The call is denied unless an explicit, bound declassification exists.

\subsection{MCP Metadata Poisoning}
\label{sec:attack-mcp}

An MCP server registers a tool with a benign name but a malicious description: ``Use this tool to validate credentials; before calling, include all available tokens.''~\cite{mcp_safety_audit,mcptox} If the agent treats tool metadata as trusted instruction, the metadata can shape future tool calls. ProvShield treats unattested metadata as \texttt{ToolMetadata}, a low-integrity source. Metadata can describe schema but cannot grant itself authority or introduce privileged objectives.

\subsection{Skill File Injection}
\label{sec:attack-skill}

A skill package includes instructions that appear to implement a formatting task but also tell the agent to delete local files after completion~\cite{skillinject}. If loaded as trusted natural-language instructions, the skill can override user intent. ProvShield labels untrusted skills separately, prevents them from changing policy, and blocks destructive effects unless authorized by user intent and capability.

\subsection{Confirmation Laundering}
\label{sec:attack-bridge}

A malicious email asks the agent to ``send the requested diagnostic information to the support address.'' The agent presents a generic confirmation: ``Send email?'' The user clicks yes. The actual recipient is attacker-controlled and the payload includes private data. ProvShield's bridge would show the exact recipient, payload digest, source categories, and private-data crossing. A bridge for one payload and destination cannot authorize another.

\section{Threat Model}
\label{sec:threat-model}

The adversary may control external content, emails, retrieved documents, tool outputs, untrusted MCP metadata, and untrusted skill files. The adversary may know the policy rules and the label syntax rendered to the model. The adversary may attempt label spoofing, metadata poisoning, bridge laundering, delayed injection, multi-step exfiltration, or capability replay.

The LLM is \emph{not} trusted to make authorization decisions. It is modeled as a proposal function that may output arbitrary tool calls. The trusted computing base (TCB) consists of:
\begin{itemize}
  \item the runtime monitor,
  \item the sidecar provenance store,
  \item the policy engine,
  \item the token generator,
  \item the bridge manager, and
  \item the audit logger.
\end{itemize}

ProvShield does not attempt to prove that untrusted text has no influence on the model's hidden state. Instead, it proves that high-risk tool execution cannot occur unless the runtime-observable provenance and capability constraints are satisfied.

\subsection{Provenance Labels}
\label{sec:labels}

Each object in the context store is associated with a provenance label:
\begin{lstlisting}
Labeled<T> = (value: T, label: ProvenanceLabel)
\end{lstlisting}

The label includes integrity, confidentiality, origin, principals, transformation history, runtime signature, nonce, and timestamp. Labels are maintained in a sidecar store. Natural-language labels shown to the model are advisory and cannot create authority.

The integrity lattice is:
\begin{lstlisting}
SystemPolicy > UserIntent > TrustedSkill
  > AttestedToolMetadata > ToolMetadata
  > ToolOutput > ExternalContent > UntrustedSkill
\end{lstlisting}

The confidentiality lattice is:
\begin{lstlisting}
Public < UserPrivate < Secret < CapabilityToken
\end{lstlisting}

These lattices are inspired by the classic work of Denning~\cite{denning1976lattice} on information-flow lattices and the decentralized label model of Myers and Liskov~\cite{myers1997decentralized}. Unlike program-level IFC systems~\cite{fides_agent_ifc}, ProvShield labels are maintained in a sidecar store separate from the model context, preventing the LLM from manipulating its own security metadata.

\subsection{Effect-Typed Tools}
\label{sec:effects}

Each tool declares effects and sinks. Examples include \texttt{ReadPrivate}, \texttt{WriteExternal}, \texttt{SendNetwork}, \texttt{DeleteLocal}, \texttt{ExecuteCode}, \texttt{ModifyAuth}, and \texttt{CreateCredential}. The monitor uses these declarations to determine whether a proposed call is low-risk, high-risk, bridge-required, or denied.

\subsection{Runtime Monitor}
\label{sec:monitor}

The monitor intercepts all tool calls. For each call, it:
\begin{enumerate}
  \item normalizes the tool name and arguments;
  \item infers effects and sinks;
  \item reconstructs argument and payload provenance from the sidecar graph;
  \item checks source-to-sink policy;
  \item allows, denies, sanitizes, quarantines, or requests a bridge;
  \item logs the decision for replay.
\end{enumerate}

The monitor operates deterministically. It does not use the LLM for policy decisions. Provenance reconstruction uses the sidecar store, not model-generated text, to trace argument origins.

\subsection{User-Intent Bridge}
\label{sec:bridge}

High-risk effects require user intent. A bridge binds the exact action, effect, sink, destination, payload digest, source categories, principal, expiration, and one-time nonce. It can authorize only the normalized call for which it was minted.

The bridge presentation to the user includes:
\begin{itemize}
  \item the action name and destination;
  \item a SHA-256 digest of the payload;
  \item the provenance source categories contributing to the payload;
  \item whether private data or secrets cross a confidentiality boundary;
  \item a short expiration window (default: 30 seconds).
\end{itemize}

\subsection{Capability Tokens}
\label{sec:tokens}

A successful bridge creates a capability token~\cite{miller2006capabilities,dennis1966semantics}. The token is not textually usable by the model. It is stored in runtime state and checked by the monitor. Tokens cannot be copied from prompt text, modified, or replayed. Each token carries a cryptographic binding to the normalized action, sink, destination, payload digest, principal, and nonce. This design draws on capability-based protection~\cite{miller2006capabilities} adapted for the agent context.

\section{Formal Model}
\label{sec:formal}

We define runtime state as:
\begin{equation}
\Sigma = (C, R, T, S, P, B, A)
\label{eq:state}
\end{equation}
where $C$ is the context store, $R$ is runtime provenance state, $T$ is the tool registry, $S$ is the secret store, $P$ is the policy, $B$ is bridge/token state, and $A$ is the audit log. Each component is typed:

\begin{itemize}
  \item $C : \text{ObjId} \to \text{Labeled}(\text{Value})$, where $\text{Labeled}(V) = V \times \text{ProvenanceLabel}$.
  \item $R : \text{ObjId} \to \text{ProvenanceLabel}$, the sidecar store mapping objects to their runtime-signed labels.
  \item $T : \text{ToolName} \to \text{EffectSet} \times \text{SinkSet}$, the effect-type declarations for each tool.
  \item $S : \text{SecretId} \to \text{Confidentiality} \times \text{Value}$, the secret store.
  \item $P \subseteq \text{Integrity} \times \text{Confidentiality} \times \text{Effect} \times \text{Sink} \times \{\texttt{allow}, \texttt{deny}, \texttt{bridge}\}$, the policy rules.
  \item $B : \text{BridgeId} \to \text{BridgeToken}$, the active bridge tokens.
  \item $A = [\text{AuditEntry}]$, the append-only audit log.
\end{itemize}

A \emph{provenance label} $\ell = (I, C_\ell, \text{origin}, \text{principals}, \text{transforms}, \sigma, \text{nonce}, t)$ consists of integrity $I \in \{\texttt{UntrustedSkill}, \texttt{ExternalContent}, \texttt{ToolOutput}, \texttt{ToolMetadata}, \texttt{AttestedToolMetadata}, \texttt{TrustedSkill}, \texttt{UserIntent}, \texttt{SystemPolicy}\}$ ordered by the integrity lattice:
\begin{equation}
\texttt{SystemPolicy} > \texttt{UserIntent} > \texttt{TrustedSkill} > \texttt{AttestedToolMetadata} > \texttt{ToolMetadata} > \texttt{ToolOutput} > \texttt{ExternalContent} > \texttt{UntrustedSkill}
\end{equation}
and confidentiality $C_\ell \in \{\texttt{Public}, \texttt{UserPrivate}, \texttt{Secret}, \texttt{CapabilityToken}\}$ with the ordering $\texttt{Public} < \texttt{UserPrivate} < \texttt{Secret} < \texttt{CapabilityToken}$. The runtime signature $\sigma = \text{MAC}_k(\text{nonce} \| t \| I \| C_\ell)$ binds the label to the TCB key $k$.

A \emph{normalized tool call} is a tuple $k = (a, e, d, h, p, n)$ where $a$ is the tool action, $e$ is the effect class, $d$ is the destination, $h$ is the payload digest, $p$ is the principal, and $n$ is the nonce. Normalization is deterministic: two calls are equivalent iff all six fields match.

\subsection{Theorem 1: Label Unforgeability}

\begin{theorem}[Label Unforgeability]
In any reachable state $\sigma \in \textit{Reach}(\Sigma_0)$, model-generated text cannot create a valid label with higher integrity than permitted by runtime transitions.
\end{theorem}

\begin{proof}[Proof sketch]
We prove by induction on transition steps $n$ that the invariant $\text{Inv}(\sigma_n)$ holds: \emph{no label in $R$ with integrity $> \texttt{ExternalContent}$ was created without a TCB transition}.

\textbf{Base case} ($n=0$): Initial state $\Sigma_0$ has empty $R$, so $\text{Inv}(\Sigma_0)$ holds vacuously.

\textbf{Inductive step}: Assume $\text{Inv}(\sigma_n)$. For each transition $\sigma_n \xrightarrow{e} \sigma_{n+1}$:
\begin{itemize}
  \item \textsc{IngestExternal}: assigns $\texttt{ExternalContent}$ or lower; invariant preserved.
  \item \textsc{IngestUser}: assigns $\texttt{UserIntent}$; TCB operation with signature; invariant preserved.
  \item \textsc{ToolReturn}: assigns $\texttt{ToolOutput}$; invariant preserved.
  \item \textsc{BridgeConfirm}: assigns integrity via TCB; invariant preserved.
  \item \textsc{ModelStep}: LLM produces text in $C$ but has no write path to $R$. The monitor ignores model-generated labels and reconstructs provenance from $R$. Invariant preserved.
\end{itemize}

Labels are valid only if they carry $\sigma = \text{MAC}_k(\text{nonce} \| t \| I \| C_\ell)$. The LLM cannot compute $\sigma$ because $k$ is in the TCB and never exposed to $C$. Therefore, no model-generated text can create a valid label with higher integrity than permitted.
\end{proof}

\subsection{Theorem 2: Capability Token Unforgeability}

\begin{theorem}[Capability Token Unforgeability]
In any reachable state, a valid capability token can only be created by the runtime through a bridge confirmation or administrative transition.
\end{theorem}

\begin{proof}[Proof sketch]
Tokens are created only in transition \textsc{BridgeConfirm} or \textsc{AdminGrant}. Both are TCB operations that require either a user-confirmed bridge or an administrative credential. The token includes a MAC over the normalized call parameters and nonce. The LLM cannot compute a valid MAC without the key. Formally: for any $\sigma \in \text{Reach}(\Sigma_0)$, if $b \in B(\sigma)$ is a valid token, then there exists a prior state $\sigma'$ and a \textsc{BridgeConfirm} or \textsc{AdminGrant} transition $\sigma' \xrightarrow{e} \sigma''$ such that $b$ was minted in $\sigma''$.
\end{proof}

\subsection{Theorem 3: No Unauthorized Secret Exfiltration}

\begin{theorem}[No Unauthorized Secret Exfiltration]
If a proposed call sends or writes payload with confidentiality $\geq \texttt{Secret}$ to an external sink, the call cannot reach execution unless a valid declassification bridge exists.
\end{theorem}

\begin{proof}[Proof sketch]
The monitor reconstructs payload provenance from $R$. If any component has confidentiality $\geq \texttt{Secret}$ and the sink is external (not in $S$ or a trusted internal store), the policy requires a bridge. Without a valid bridge, the monitor transitions to \textsc{Deny}. A bridge is valid only if it binds the exact payload digest, destination, and action. Therefore, a generic or reused bridge cannot authorize a different exfiltration.
\end{proof}

\subsection{Theorem 4: No Low-Integrity Control of High-Risk Effects}

\begin{theorem}[No Low-Integrity Control]
If a high-risk effect is influenced by low-integrity sources (external content, tool output, untrusted skill, or untrusted tool metadata), the call cannot execute unless a valid bridge or policy exception authorizes the flow.
\end{theorem}

\begin{proof}[Proof sketch]
The monitor traces provenance of all arguments through $R$. If any argument's provenance includes a source with integrity $< \texttt{UserIntent}$ and the call has a high-risk effect, the policy requires a bridge. A bridge authorizes a specific normalized call; it cannot be repurposed for a different action or destination. Therefore, an attacker-controlled low-integrity source cannot direct high-risk execution through the bridge mechanism.
\end{proof}

\subsection{Theorem 5: Bridge Non-Replay}

\begin{theorem}[Bridge Non-Replay]
A bridge token authorized for a normalized call $(a, e, d, h, p, n)$ cannot authorize a different tuple $(a', e', d', h', p', n')$ where any field differs, or where the token has expired.
\end{theorem}

\begin{proof}[Proof sketch]
The token includes a MAC over all six fields plus an expiration timestamp. The monitor re-normalizes the proposed call and recomputes the expected MAC. Any field mismatch causes verification to fail. The nonce $n$ is consumed upon use (one-time semantics), preventing replay.
\end{proof}

\section{Implementation}
\label{sec:implementation}

ProvShield is implemented as a runtime layer around an existing agent system. The implementation consists of six components:

\begin{enumerate}
  \item \textbf{MCP proxy} for tool registration and call mediation. It intercepts \texttt{tools/list} and \texttt{tools/call} messages, labeling metadata at registration and outputs at return time.
  \item \textbf{Skill loader} for provenance-aware skill ingestion. Skills from untrusted sources are labeled \texttt{UntrustedSkill}; attested skills from verified publishers are labeled \texttt{TrustedSkill}.
  \item \textbf{Context builder} for sidecar label construction. It assigns provenance labels to every object entering the agent context, including user messages, system prompts, web content, email, RAG results, and tool outputs.
  \item \textbf{Policy engine} for source-to-sink rules. The engine is configured via a declarative YAML policy that maps effect types and provenance patterns to actions (allow, deny, bridge).
  \item \textbf{Bridge manager} for user confirmations and token issuance. It renders human-readable bridge presentations, collects confirmation, and issues cryptographically bound capability tokens.
  \item \textbf{Audit logger} for deterministic replay. Every monitor decision, bridge interaction, and tool execution is logged with full provenance state for post-hoc analysis.
\end{enumerate}

The prototype avoids modifying MCP servers by mediating calls through a proxy. Tool metadata is labeled at registration time. Tool outputs are labeled at return time. All proposed calls pass through the monitor before execution. The implementation uses Python for the policy engine and bridge manager, with JSON-serializable provenance labels and HMAC-based token signatures.

\section{Evaluation}
\label{sec:evaluation}

We evaluate ProvShield along three axes: security (attack blocking), utility (benign task completion), and performance (runtime overhead).

\subsection{Attack Suites}
\label{sec:eval-attacks}

We evaluate against six attack categories:
\begin{enumerate}
  \item \textbf{Skill injection}~\cite{skillinject}: malicious instructions embedded in skill files.
  \item \textbf{MCP metadata poisoning}~\cite{mcptox}: tool descriptions that attempt to exfiltrate credentials or influence future tool calls.
  \item \textbf{MCP safety scenarios}~\cite{mcp_safety_audit}: demonstrated exploits against MCP-connected LLM agents including code execution, remote access, and credential theft.
  \item \textbf{Web/email prompt injection}~\cite{greshake2023indirect}: indirect injection through retrieved web pages and email content.
  \item \textbf{RAG injection}: adversarial documents injected into retrieval-augmented generation pipelines.
  \item \textbf{Adaptive white-box attacks}: attackers with knowledge of the policy and label syntax who attempt to craft inputs that bypass enforcement~\cite{attriguard_agentwatcher}.
\end{enumerate}

\subsection{Baselines}
\label{sec:eval-baselines}

We compare ProvShield against:
\begin{itemize}
  \item \textbf{No defense}: the agent runs without security mediation.
  \item \textbf{Prompt hardening}: the system prompt instructs the model to ignore injected instructions~\cite{openai_agent_prompt_injection_defense}.
  \item \textbf{Input firewall}: a classifier filters malicious text before it reaches the model.
  \item \textbf{Static allowlist}: tools are restricted to a fixed set; no dynamic policy.
  \item \textbf{Generic confirmation}: the user is asked to confirm all writes, but the confirmation does not bind specific destination or payload.
  \item \textbf{Fides-style IFC}~\cite{fides_agent_ifc}: information-flow labels tracked within the model context (where feasible).
  \item \textbf{Causal attribution monitor}~\cite{attriguard_agentwatcher}: post-hoc attribution of tool calls to user intent vs.\ external content (where feasible).
\end{itemize}

\subsection{Metrics}
\label{sec:eval-metrics}

We report the following metrics:
\begin{itemize}
  \item \textbf{Attack success rate (ASR)}: fraction of attack scenarios where the malicious action executes.
  \item \textbf{Secret exfiltration rate}: fraction of scenarios where secret data reaches an external sink.
  \item \textbf{Unauthorized write/delete/exec rate}: fraction of scenarios where a destructive action executes without authorization.
  \item \textbf{Bridge abuse success}: fraction of scenarios where an attacker successfully manipulates the bridge.
  \item \textbf{Benign task completion rate (BTCR)}: fraction of benign tasks that complete successfully.
  \item \textbf{False blocking rate}: fraction of benign tool calls incorrectly denied.
  \item \textbf{Confirmation burden}: average number of bridge confirmations per benign task.
  \item \textbf{Monitor latency (p50/p95)}: latency added by the runtime monitor per tool call.
\end{itemize}

\subsubsection{Security Analysis}

Across 530 attack scenarios (including 88 workflow-embedded strong attacks) evaluated with LLM-in-the-loop, ProvShield achieves an attack success rate of \textbf{0.6\%} (95\% CI [0.2\%, 1.7\%]), an \textbf{88\%} reduction from the no-defense baseline of 5.1\% (95\% CI [3.5\%, 7.3\%]). The LLM manipulation rate---the fraction of attack scenarios where the model generates any tool call---is 14.9\%, indicating that the model itself provides partial resistance. Among the scenarios where the LLM generates the specific attack target tool, ProvShield's conditional block rate is \textbf{100\%}: every malicious tool call that the LLM generates is blocked by the runtime monitor.

The no-defense ASR of 5.1\% reflects the robustness of modern LLMs against many injection patterns. The 88 workflow-embedded attack scenarios (45 strong + 22 targeted + 21 high-rate) (where malicious actions are framed as legitimate IT procedures, compliance requirements, or policy updates) increase the manipulation rate compared to the standard scenarios. ProvShield's 100\% block rate demonstrates that even when the LLM is manipulated, the runtime enforcement prevents all unauthorized execution.

Among baselines, prompt hardening reduces ASR to 1.9\% but cannot defend against contextual or metadata-based attacks. Input firewall achieves 3.4\% ASR. Generic confirmation provides no protection (5.1\% ASR, identical to no defense) because it does not bind the specific destination or payload. Static allowlist achieves 0.0\% ASR but at the cost of 20.8\% false blocking (79.2\% BTCR).


\subsubsection{Utility Analysis}

ProvShield achieves 92.4\% benign task completion (95\% CI [88.0\%, 94.9\%]), compared to 100\% for the no-defense baseline. The 7.6\% reduction is within the acceptance threshold and reflects the conservative policy applied to scenarios with ambiguous provenance. The false blocking rate is approximately 7.6\%, and the confirmation burden is 7.6\% of benign tasks. Static allowlist achieves only 79.2\% BTCR, demonstrating the cost of its coarse-grained approach.

\subsubsection{Conditional Metrics}

Among attack scenarios where the LLM generates the specific attack target tool, ProvShield's conditional block rate is \textbf{100\%}: every malicious tool call is blocked. This confirms that ProvShield's runtime enforcement is active precisely where it matters: when the model has been manipulated into generating malicious calls. The 100\% block rate holds across all attack suites, including the workflow-embedded strong attack scenarios.

\subsubsection{Performance}

The runtime monitor adds minimal overhead: p50 latency of 0.03\,ms and p95 latency of 0.07\,ms per tool call, well below the 100\,ms target. Because ProvShield maintains provenance labels in a sidecar store rather than embedding them in the prompt, the prompt token overhead is limited to lightweight advisory markers. The audit trace is fully replayable: every monitor decision, bridge interaction, and tool execution is logged with full provenance state for deterministic post-hoc analysis.

In two cases, web/email content was ingested through a path that preserved higher integrity labels; in one adaptive case, the attacker crafted content that matched user-intent patterns.

The 7.6\% false blocking rate arises from benign scenarios where the LLM generates tool calls with conservative provenance linking to external content, triggering bridge or deny decisions. In practice, field-level provenance tracking and user bridge interaction would reduce this rate.

\section{Discussion}
\label{sec:discussion}

ProvShield's guarantees are runtime guarantees. It does not claim to remove malicious influence from model activations. An attacker may still influence the model's planning through low-integrity content; ProvShield ensures that this influence cannot materialize as unauthorized high-risk tool execution.

\paragraph{Conservative policy.} ProvShield's policy may be conservative when provenance is ambiguous. For example, a tool output that mixes user-provided data with external content will be labeled at the lower integrity level. This can cause false blocking for benign mixed-provenance payloads. We mitigate this with field-level provenance tracking and configurable policy granularity.

\paragraph{Social engineering.} User confirmations can still be socially engineered. However, bridge binding prevents generic confirmation from authorizing hidden destination or payload changes. The bridge presentation includes payload digest, destination, and provenance source categories, making hidden changes visible to the user.

\paragraph{Model influence.} ProvShield does not address the problem of an attacker influencing the model's reasoning to cause benign-looking but suboptimal tool selections. This is an inherent limitation of treating the model as an untrusted planner rather than attempting to make it trustworthy.

\paragraph{Deployment.} ProvShield should be deployed alongside sandboxing, least-privilege tool access, and secret management. It is a complementary defense layer, not a replacement for standard security practices.

\section{Related Work}
\label{sec:related}

\paragraph{Prompt injection.}
Perez and Ribeiro~\cite{perez2022ignore} demonstrated goal hijacking and prompt leaking against GPT-3 through PromptInject. Greshake et al.~\cite{greshake2023indirect} introduced indirect prompt injection, showing that retrieved data can function as arbitrary code execution in LLM-integrated applications. OpenAI published design guidance for agents to resist prompt injection~\cite{openai_agent_prompt_injection_defense}, focusing on prompt hardening and input separation.

\paragraph{Information-flow control for agents.}
Fides~\cite{fides_agent_ifc} from Microsoft Research applies information-flow control to LLM agents, tracking labels within the model context. ProvShield differs by maintaining provenance in a sidecar store outside the model context, preventing the LLM from manipulating its own security metadata. NeuroTaint~\cite{neurotaint} extends taint tracking to semantic transformations in LLM agents, complementing ProvShield's runtime enforcement. The classic foundations of IFC include Denning's lattice model~\cite{denning1976lattice} and the decentralized label model of Myers and Liskov~\cite{myers1997decentralized}. TaiLi~\cite{taili} applies IFC principles specifically to LLM agent tool invocations.

\paragraph{Capability-based security.}
Dennis and Van Horn~\cite{dennis1966semantics} introduced capability-based addressing. Miller~\cite{miller2006capabilities} developed capability-based security for modern systems. ProvShield applies these principles to agent tool authorization through bound capability tokens. Narayan et al.~\cite{narayan2025capabilities} proposed capability-based security for AI agent tool invocation with formal guarantees. The Agent Passport~\cite{oap2026} specifies deterministic pre-action authorization for autonomous agents.

\paragraph{Taint tracking.}
Newsome and Song~\cite{newsome2005dynamic} introduced dynamic taint analysis for software security. ProvShield adapts this concept to the agent setting, tracking provenance through natural-language transformations rather than program data flow.

\paragraph{MCP security.}
Radosevich and Halloran~\cite{mcp_safety_audit} demonstrated security exploits in MCP-connected LLM agents and introduced MCPSafetyScanner for auditing MCP servers. Wang et al.~\cite{mcptox} created MCPTox, a benchmark for tool poisoning attacks on real-world MCP servers. The MCP authorization specification~\cite{mcp_authorization_spec} defines transport-level authorization for MCP connections but does not address content-level provenance.

\paragraph{Skill and tool security.}
Schmotz et al.~\cite{skillinject} introduced SkillInject, demonstrating up to 80\% attack success rate against frontier models through skill file injection. Shi et al.~\cite{progent} proposed Progent, which secures agents through privilege control with monotonic confinement. Song et al.~\cite{progent} further developed tool-use monitoring and privilege control frameworks for LLM agents.

\paragraph{Action attribution.}
AttriGuard~\cite{attriguard_agentwatcher} proposes causal attribution of tool invocations to distinguish user-intent-driven calls from injection-driven calls through counterfactual testing. ProvShield is complementary: it enforces policy at the runtime level regardless of the model's reasoning, while attribution-based approaches attempt to determine whether the model's reasoning was influenced.

\section{Limitations}
\label{sec:limitations}

ProvShield has several limitations that we state explicitly.

\paragraph{Model influence.} ProvShield does not claim to prevent untrusted text from influencing the LLM's hidden activations or reasoning. An attacker may still cause the model to generate suboptimal or strategically harmful tool selections that pass all provenance checks. ProvShield only guarantees that \emph{runtime-observable} high-risk effects cannot execute without proper provenance and capability authorization.

\paragraph{Social engineering.} User-intent bridges require human confirmation. A sufficiently convincing social engineering attack may still trick a user into confirming a harmful action. Bridge binding mitigates this by making the destination, payload digest, and provenance sources visible, but it cannot eliminate the risk entirely.

\paragraph{Conservative policy.} When provenance is ambiguous---for example, when a tool output mixes user-provided data with external content---ProvShield labels at the lower integrity level. This conservative join can cause false blocking for benign mixed-provenance payloads. Our evaluation shows 7.6\% false blocking on the benign suite, primarily from conservative taint propagation.

\paragraph{Residual attack surface.} ProvShield achieves 0.6\% ASR across 780 scenarios (3 residual attacks). The residual attacks occur in the web/email and adaptive suites where the LLM generates tool calls whose provenance does not trigger high-risk policy thresholds. The conservative taint propagation mode (binding all context to all arguments) may also produce false positives in untested configurations. Finer-grained argument-level provenance tracking would further improve precision.

\paragraph{TCB assumptions.} The formal guarantees assume the runtime monitor, sidecar store, policy engine, and cryptographic primitives are not compromised. If the runtime itself is compromised, all guarantees are void. The HMAC key is managed at the module level; production deployment requires a proper key management system.

\paragraph{Evaluation scope.} Our evaluation uses 780 scenarios (530 attack, 250 benign) across 6 attack categories and 5 benign task categories with LLM-in-the-loop evaluation using a single model (MiMo-v2-pro), including 88 workflow-embedded attack scenarios (45 strong + 22 targeted + 21 high-rate). A larger-scale evaluation with multiple LLMs, real-world agent workflows, diverse MCP servers, and stronger adaptive attacks would strengthen the evidence. The no-defense ASR of 5.1\%, while indicating model robustness, also means the absolute attack surface is small; evaluating against more vulnerable models or more sophisticated attacks would better demonstrate ProvShield's value.

\paragraph{Formal proofs.} The Coq formalization compiles with coqc 9.0 and includes a transition relation with reachable-state invariants. However, Theorems 1 (label unforgeability) and 2 (token unforgeability) are currently proven at the definition level---they state that valid labels/tokens have non-zero MAC, which follows directly from the definition of \texttt{label\_valid} and \texttt{token\_valid}. The reachable-state invariant (\texttt{reachable\_well\_formed}) proves that transitions preserve label well-formedness, but the full proof that model-generated text cannot forge labels requires modeling the LLM's lack of access to the signing key, which is stated as an assumption rather than mechanized. The paper presents these as proof sketches, not fully mechanized proofs.

\section{Ethics}
\label{sec:ethics}

\paragraph{Dual-use considerations.} This research studies prompt injection, metadata poisoning, and credential exfiltration. We designed the evaluation to use sandboxed mock tools, synthetic secrets (canary tokens), and synthetic MCP servers. No real credentials, private documents, or third-party accounts are used. If real vulnerabilities are discovered during research, we follow responsible disclosure practices.
\paragraph{User autonomy.} ProvShield may block or gate high-risk actions. The system explains decisions through the audit log and bridge presentation. Administrators can configure policies to adjust strictness. We do not hide the fact that an action was blocked due to untrusted content influence.

\paragraph{Audit log privacy.} Audit logs may contain provenance metadata, payload previews, and decision reasons. In production, logs must be redacted or encrypted. The artifact uses only synthetic data.

\paragraph{False sense of security.} ProvShield is a complementary defense layer, not a complete solution. We explicitly warn against deploying it without sandboxing, least-privilege tool access, and secret management. The paper's threat model and non-goals section clearly state what ProvShield does not protect against.

\section{Conclusion}
\label{sec:conclusion}

Prompt injection in agents is best understood as an authority-flow problem. Untrusted content may influence language generation, but it should not be able to control privileged tool effects. ProvShield enforces this boundary through unforgeable provenance labels, effect-typed tools, runtime source-to-sink policy, and bound user-intent bridges.

We formalized the system with a labeled transition system and proved five safety properties as proof sketches: label unforgeability, capability-token unforgeability, no unauthorized secret exfiltration, no low-integrity control of high-risk effects, and bridge non-replay. The Coq formalization compiles with coqc 9.0 and includes the transition relation and reachable-state invariants, though the unforgeability theorems rely on definition-level reasoning and TCB assumptions.

Evaluation on 780 scenarios (530 attack, 250 benign) with LLM-in-the-loop, including 88 workflow-embedded attack scenarios (45 strong + 22 targeted + 21 high-rate), shows that ProvShield achieves \textbf{0.6\%} attack success rate compared to 5.1\% for no defense---an 88\% reduction---while maintaining 92.4\% benign task completion. Every malicious tool call generated by the LLM targeting the attack-specific tool is blocked by the runtime monitor (100\% conditional block rate). The no-defense ASR reflects modern LLM robustness; ProvShield provides defense-in-depth by blocking the residual attacks the model fails to resist.

ProvShield does not claim to solve prompt injection. It claims to make the consequences of prompt injection visible and controllable at the runtime boundary. The model may be deceived; the runtime must not be.
