# Provenance Label and Policy Specification

## 1. Design principle

Labels are not plain text. They are runtime-side metadata bound to objects in the context store, tool registry, argument graph, bridge state, and audit log.

The model may see rendered labels such as `[ExternalContent: web]`, but the monitor ignores any label text generated by the model.

## 2. Label object

```ts
type Labeled<T> = {
  value: T;
  label: ProvenanceLabel;
}

type ProvenanceLabel = {
  integrity: IntegrityLabel;
  confidentiality: ConfidentialityLabel;
  origin: OriginDescriptor;
  principals: Principal[];
  transform_history: TransformStep[];
  runtime_signature: string;
  nonce: string;
  created_at: string;
}
```

## 3. Integrity lattice

```text
SystemPolicy
  > UserIntent
  > TrustedSkill
  > AttestedToolMetadata
  > ToolMetadata
  > ToolOutput
  > ExternalContent
  > UntrustedSkill
```

### Notes

- `ToolMetadata` is not trusted by default.
- `AttestedToolMetadata` requires registry approval, signature verification, version pinning, or administrator approval.
- `ToolOutput` remains low-integrity even if the tool itself is allowed.
- `ExternalContent` may be useful evidence, but cannot create new privileged objectives.
- `UntrustedSkill` is below external content because it may carry executable workflow instructions.

## 4. Confidentiality lattice

```text
Public < UserPrivate < Secret < CapabilityToken
```

### Notes

- `Secret` includes API keys, OAuth tokens, cookies, credentials, private tokens, environment variables.
- `CapabilityToken` is treated as the highest confidentiality class.
- Tool output may become `UserPrivate` or `Secret` depending on the source.

## 7. Core policy rules

### P1: No low-integrity control of high-risk effects

```text
If effect in {WriteExternal, DeleteExternal, SendNetwork, ExecuteCode, ModifyAuth, CreateCredential, FinancialAction}
and argument provenance includes low-integrity source
then deny or require UserIntentBridge.
```

### P2: No secret exfiltration

```text
If payload confidentiality >= Secret
and sink in {ExternalWriteSink, NetworkSendSink}
and no valid declassification bridge
then deny.
```

### P3: Tool metadata cannot authorize itself

```text
ToolMetadata cannot increase its own integrity,
create new tool permissions,
or cause policy updates.
```

### P4: Tool output cannot create new objective

```text
ToolOutput may supply facts but cannot introduce high-risk goals.
A high-risk goal must originate from UserIntent or SystemPolicy.
```

### P5: Capability token unforgeability

```text
Only runtime can mint CapabilityToken.
Model-generated strings never count as valid tokens.
```

### P6: Bridge binding

```text
A bridge authorizes exactly one action class,
one sink,
one destination,
one payload digest,
one expiry,
one nonce,
one principal.
```

### P7: Label preservation

```text
Transform output label = join(input labels) adjusted by approved sanitizer/declassifier rules.
```

## 8. Label propagation examples

### Example 1: Summarizing a webpage

```text
Input: ExternalContent/Public
Operation: summarize
Output: ExternalContent/Public
Allowed to display locally.
Not allowed to send externally without bridge if destination was suggested by page.
```

### Example 2: Summarizing private email

```text
Input: UserPrivate
Operation: summarize
Output: UserPrivate
Allowed to show to user.
Not allowed to post to Slack without bridge.
```

### Example 3: Webpage asks agent to email token

```text
Input A: ExternalContent/Public with malicious instruction
Input B: Secret
Proposed sink: SendNetwork
Decision: deny, because Secret -> ExternalSink and ExternalContent controls action.
```

### Example 4: User explicitly asks to email a report

```text
Input A: UserIntent: "Email this report to alice@example.com"
Input B: UserPrivate report
Proposed sink: SendNetwork to alice@example.com
Decision: require bridge if report contains private data; allow after bound confirmation.
```
